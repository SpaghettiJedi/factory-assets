### Decisions not in the spec

- `[product]` The retry allowance resets every review round rather than once per run. The WorkOrder asked for "one bounded retry"; as built, a reviewer that keeps misformatting can be re-run once in each of the three review rounds, so the worst case is six paid reviewer sessions per pull request instead of the previous three. If you want a single retry for the whole run, that needs a follow-up decision.
- `[technical]` The new recurrence gate verifies the review stage by matching exact phrases in its source code. This makes the gate deterministic, but a future refactor that renames things while preserving behavior will turn the whole quality gate red until the gate is updated in the same change.
- `[technical]` The gate's self-test covers six recurrence classes, three more than the WorkOrder required (unbounded retry, publishing malformed output, and per-attempt round events). Extra coverage, no downside.

### Manual checks for the operator

- After the next real review run that hits a malformed reviewer response, open the run report and confirm the evidence reads sensibly: one round entry per logical round, with the protocol attempts and the raw invalid outputs visible only in the local journal, never in a GitHub comment. If this check feels routine after one or two runs, it is already largely covered by the new gate and can be dropped.

### Non-obvious behaviors

- [undeclared] The reviewer-retry budget resets each round, so a persistently misformatting reviewer can consume up to twice the reviewer sessions of the old design across a full three-round run.
- [undeclared] In the behavior parser, a numbered line introducing a "what" field starts a new declaration at any indentation depth — even in the middle of another declaration's descriptive text. An agent that quotes an example declaration inside its reasoning will have its entry silently split at that point.
- [undeclared] The recurrence gate is coupled to the literal source text of the review stage, so behavior-preserving refactors of that file will fail `make check` until the gate is co-updated.
- [undeclared] Raw malformed reviewer outputs are stored verbatim and in full in the local evidence journal; nothing truncates them.

### Be aware

- Cost: the failure mode this WorkOrder handles (a reviewer vendor that stops following the output format) now bills an extra reviewer session per affected round before failing closed. That is the intended trade — a protocol hiccup no longer wastes a code-remediation round or reaches your review surface — but it shows up as higher reviewer spend during any period when a vendor's formatting is flaky.
- The fail-closed path leaves the pull request open with any remediation commits already pushed; the issue arrives at you labeled for human attention with the reason recorded, and no comment for the failed round appears on the pull request. That silence on the PR is by design.

### Required changes

- Complete the final structured behavior declaration in the run's progress file (currently "None"), declaring each of the four undeclared behaviors listed above: the per-round reset of the reviewer-retry budget, the parser's any-indentation rule for numbered "what" lines splitting declarations, the recurrence gate's coupling to the review stage's exact source text, and the verbatim untruncated storage of malformed reviewer outputs in the evidence journal. No code changes are required unless the operator rejects the per-round retry budget, in which case a global retry cap should be filed as a separate WorkOrder.
