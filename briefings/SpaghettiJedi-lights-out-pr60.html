## Operator Briefing

### Scope: done / not done

**✅ Test suite covers all decision logic** — HEAD-change detection, dirty-tree deferral, once-mode bypass, and exit-after-pass sequencing are tested via pure functions over daemon state snapshots (no live daemon required).

**✅ make check passes** — All gates green.

### Decisions not in the spec

**[technical]** Decision functions (`decideSelfRestart`, `decideAfterPass`) are separate pure functions rather than inline checks — this structure makes the "exit after pass" ordering testable without running a live daemon, matching the WorkOrder's test requirement.

**[technical]** Dirty-restart logging uses per-sha deduplication (`dirtyRestartLoggedForSha`) — logs once when the checkout first advances to a dirty commit, silent on subsequent passes while it remains dirty at that same sha.

**[technical]** Dirty detection via `git status --porcelain` output length — catches all change types (staged, unstaged, untracked), but could be slow if the daemon's checkout accumulates thousands of untracked files.

### Manual checks for the operator

**1. Restart flow via launchd** [recurs for any process-supervision work]  
With the daemon running, push a commit to its checkout, wait for the next pass (poll interval from logs). Success: daemon logs "factory daemon restarting on new build <sha>", exits cleanly, and launchd brings it back on the new code within seconds.

**2. Daemon-up notification suppression post-restart** [recurs if notification logic changes]  
Trigger a restart (per check 1), watch for daemon-up notifications. Success: the restart itself notifies once ("factory daemon restarting..."), but no daemon-up notification fires when launchd restarts it — existing suppression still applies.

**3. Dirty checkout defers restart** [one-time unless dirty-handling changes]  
With the daemon running, make an uncommitted change in its checkout, push a commit, wait for a pass. Success: daemon logs "checkout advanced to <sha> but working tree is dirty; not restarting" once and continues on old code until the tree is clean.

### Be aware

The daemon now restarts on *any* checkout advancement, including non-code changes (docs, tests, workflows). During active nucleus development, expect restarts every few minutes — each briefly pauses work (current pass finishes, then downtime until launchd recovers). This is by design to prevent stale-code incidents, but rapid merges create churn. If the checkout stays dirty (operator has uncommitted work), the daemon will *not* restart — it could run stale code indefinitely until the tree is cleaned. No notification fires to remind the operator of this state beyond the one-time log line.
